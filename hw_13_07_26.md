Почему контейнер остановился?
Команда docker attach подключает терминал к основному процессу (PID 1) внутри контейнера. В случае с официальным образом Nginx, это и есть сам процесс веб-сервера, работающий на переднем плане. Нажатие Ctrl-C отправляет сигнал прерывания (SIGINT) этому основному процессу.
Когда PID 1 завершает работу, Docker автоматически останавливает весь контейнер, так как его главная задача выполнена.

Суть возникшей проблемы:
При создании контейнера (командой docker run) мы жестко задали правило проброса портов: трафик с порта хоста 8080 перенаправляется на порт 80 внутри контейнера. Это правило неизменно на протяжении всего жизненного цикла контейнера. 
После того как мы изменили конфигурацию Nginx внутри контейнера, он перестал слушать порт 80 и начал слушать порт 81. В результате Docker-прокси по-прежнему стучится на порт 80 внутри контейнера, но там его никто не принимает, что приводит к ошибке соединения.

### Объяснение результата
Мы использовали механизм **Bind Mounts** (ключ `-v $(pwd):/data`). Этот механизм создает прямую связь между директорией на хост-машине и директорией внутри контейнера на уровне файловой системы ОС. 
Поскольку оба контейнера (`centos-t4` и `debian-t4`) и хост-машина обращаются к одной и той же физической папке на диске, любые изменения (создание, редактирование, удаление файлов), сделанные из любой из этих трех точек, мгновенно становятся видны в двух других. Файлы не копируются, а монтируются как есть.

При выполнении команды docker compose up -d без явного указания файла, Docker Compose ищет конфигурационные файлы в текущей директории в строгом порядке приоритета: compose.yaml имеет высший приоритет перед docker-compose.yaml.
Поэтому был запущен только сервис из compose.yaml (Portainer), а файл docker-compose.yaml был проигнорирован.

Суть предупреждения:
При удалении файла compose.yaml Docker Compose при запуске опирается только на оставшийся файл docker-compose.yaml, в котором описан только сервис registry.
Однако контейнер portainer физически продолжает работать в системе. Docker называет такие контейнеры "orphaned" (осиротевшими), так как они запущены в рамках этого проекта, но больше не описаны в его текущей конфигурации.
Предупреждение предлагает использовать флаг --remove-orphans для их автоматической очистки.

<img width="1517" height="340" alt="изображение" src="https://github.com/user-attachments/assets/5dfcceb4-493d-4aa7-a576-0d192ac51bbf" />

   ### Файл compose.yaml до его удаления
   ```yaml
   include:
     - docker-compose.yaml
   version: "3"
   services:
     portainer:
       network_mode: host
       image: portainer/portainer-ce:latest
       volumes:
         - /var/run/docker.sock:/var/run/docker.sock
   ```
